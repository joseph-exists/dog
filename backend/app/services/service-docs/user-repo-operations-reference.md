# User Repo Operations Reference

## Audience

Operations engineer responsible for extending, diagnosing, or restoring the platform-managed user repo import path backed by `gittin`/Gogs.

## Scope

This reference covers the operational surface for importing external HTTPS repositories into platform-managed `dog/*` repos through the backend.

It does not cover:

- direct user access to Gogs
- shadow repo flows except where they share infrastructure
- frontend implementation details

## System Model

### Ownership Model

- Platform users do not own Gogs repos directly.
- Imported user-visible repos live under the `dog` org.
- Repo names are generated by the backend as `dog/{slug}-{short_id}`.
- The backend is the system of record for authorization and repo metadata.

### Request Path

1. Client calls `POST /api/v1/user-repos/`.
2. Backend validates `source_repo_url`.
3. Backend creates a `user_repos` row.
4. Backend attempts Gogs repo migration/import using service credentials.
5. Backend updates import status to `ready` or `failed`.
6. Client reads repo state via `GET /api/v1/user-repos/` or `GET /api/v1/user-repos/{repo_id}`.

## Required Configuration

### Backend

- `USER_REPO_GOGS_BASE_URL`
- `USER_REPO_GOGS_TOKEN`
- `USER_REPO_GOGS_ORG`
- `USER_REPO_GOGS_TIMEOUT_SECONDS`
- `USER_REPO_DEFAULT_BRANCH`

### Expected Defaults

- Gogs base URL points at `gittin`
- user repo org is `dog`
- default branch is `main`

## Database Contract

`user_repos` now includes:

- `source_repo_url`
- `source_branch`
- `import_status`
- `import_error`
- `imported_at`

Operational meaning:

- `pending`: record exists but import has not started
- `importing`: backend has accepted work and is attempting provisioning/import
- `ready`: managed repo exists and metadata has been synchronized
- `failed`: import did not complete; inspect `import_error`

## Gogs API Assumptions

### Lookup

The backend checks:

- `GET /api/v1/repos/{org}/{repo_name}`

### Empty Repo Creation Fallback

For non-import use cases:

- `POST /api/v1/org/{org}/repos`

### Import/Migration

Primary path:

- `POST /api/v1/repos/migrate`

Fallback targeting strategies attempted by backend:

- payload with `repo_owner`
- payload with `uid` if org lookup returns an id

Org id lookup attempts:

- `GET /api/v1/orgs/{org}`
- `GET /api/v1/org/{org}`

## Operational Caveat

The backend contract expresses a `main`-branch-only import intent, but Gogs migration/import may not give exact shallow single-branch semantics.

Implication:

- A successful `ready` state means the platform import workflow succeeded.
- It does not, by itself, prove that the underlying Git object graph is strictly limited to one branch or zero history.
- If strict snapshot semantics are required, the implementation will need a different backend import strategy.

## Normal Success Indicators

### API Layer

- `POST /api/v1/user-repos/` returns `201`
- response body includes:
  - `import_status = ready`
  - `gogs_repo_id`
  - `gogs_full_name`

### Database Layer

- row exists in `user_repos`
- `gogs_repo_name` is populated
- `import_error` is null
- `imported_at` is set

### Gogs Layer

- repo exists under `dog`
- repo name matches `gogs_repo_name`
- service token can look it up

## Failure Modes

### Backend Not Configured

Symptoms:

- repo record created
- `import_status = failed`
- `import_error` mentions provisioning not configured

Likely causes:

- missing `USER_REPO_GOGS_BASE_URL`
- missing `USER_REPO_GOGS_TOKEN`

### Gogs Auth Failure

Symptoms:

- import fails quickly
- `import_error` includes non-2xx Gogs API response

Likely causes:

- invalid token
- token lacks org/repo permissions
- token belongs to wrong service account

### Org Resolution Failure

Symptoms:

- import fails during migration call
- possible `404`, `400`, or `422` from migrate path

Likely causes:

- `dog` org missing
- org name mismatch
- Gogs version/API shape mismatch around migration targeting

### Source Repo Reachability Failure

Symptoms:

- import row exists but lands in `failed`
- Gogs may report inability to fetch remote

Likely causes:

- source repo does not exist
- source repo is private and inaccessible
- outbound network path from `gittin` cannot reach source host
- TLS or DNS problems inside container network

### Data Drift

Symptoms:

- Gogs repo exists but backend row lacks metadata
- repeated requests produce inconsistent state

Likely causes:

- partial migration success with interrupted backend response cycle
- manual repo changes in Gogs
- stale token or org configuration after repo creation

## Triage Workflow

### 1. Confirm Backend Config

Check the effective values for:

- `USER_REPO_GOGS_BASE_URL`
- `USER_REPO_GOGS_ORG`
- token presence

### 2. Inspect API Result

Look at the returned repo record:

- `import_status`
- `import_error`
- `gogs_repo_name`
- `source_repo_url`

### 3. Inspect Database Row

Confirm persisted state in `user_repos`.

Key fields:

- `source_repo_url`
- `import_status`
- `import_error`
- `gogs_repo_id`
- `gogs_full_name`

### 4. Inspect Gogs

Check whether the target repo exists in `dog`.

If it exists:

- compare backend metadata with actual Gogs repo state

If it does not:

- inspect service logs for migration failure details

### 5. Inspect Container and Service Logs

Priority logs:

- backend app logs
- `gittin` container logs
- Gogs logs under [`gittin/data/log`](/home/josep/dog/gittin/data/log)

### 6. Verify Org and Token

Confirm:

- token can see `dog`
- token can create/import repos under `dog`
- org id lookup succeeds for the deployed Gogs version

## Safe Recovery Strategies

### If Repo Row Failed and No Gogs Repo Exists

- fix config or external cause
- re-run import through normal application flow

### If Gogs Repo Exists but Backend Row Is Failed or Incomplete

- inspect whether repo lookup now succeeds through backend service
- prefer backend reconciliation over manual database edits when possible
- avoid renaming the repo in Gogs

### If Source Repo Is Invalid

- leave failed record visible for audit/debugging
- create a new import with corrected source URL rather than mutating historical provenance silently

## Extension Guidance

When extending this implementation:

- keep all user access mediated through backend APIs
- do not expose Gogs credentials or direct forge workflows to users
- prefer adding explicit backend status fields over inferring state in the frontend
- treat org/repo naming as stable once generated
- preserve `dog` org ownership model

## Implementation Notes For Critical Fixes

### If Gogs API Shape Changes

The first place to adjust is [`user_repo_service.py`](/home/josep/dog/backend/app/services/user_repo_service.py):

- `_get_repo`
- `_migrate_repo`
- `_build_migration_payloads`
- `_get_org_id`
- `_raise_api_error`

### If Import Needs Stronger Git Guarantees

Do not keep stretching the current migration call beyond what Gogs actually supports. Introduce a backend-managed import pipeline that:

- fetches external source directly
- materializes only the desired snapshot
- pushes platform-owned result into managed repo

That is a different operational model and should be treated as a deliberate implementation change, not a small patch.

## Minimum Smoke Checks After Changes

1. Create a new import from a known public HTTPS repository.
2. Confirm `201` response from `POST /api/v1/user-repos/`.
3. Confirm repo row reaches `ready`.
4. Confirm repo exists in `dog`.
5. Confirm list and detail endpoints both return the same record state.
